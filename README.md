# RAG Pipeline

A from-scratch implementation of a Retrieval-Augmented Generation (RAG) pipeline, built as part of the Factored Jed-AI mission series.

## Assignment

Build an end-to-end RAG pipeline over a corpus of your choice that:

- Ingests and chunks documents
- Embeds them using at least two different embedding models
- Stores them in a vector store
- Implements semantic search wired to an LLM to produce grounded, cited responses
- Experiments with at least two chunking strategies and compares retrieval quality between them

All responses must include source citations. If the answer isn't in the corpus, the system must say so.

## What we built

A board game rules assistant. Upload a PDF rulebook, ask questions, get cited answers. The pipeline runs fully locally via Ollama.

**Ingestion & chunking**

PDF rulebooks are parsed to Markdown using LlamaCloud (agentic tier for layout fidelity). Two chunking strategies are implemented and compared:

| Strategy | Description |
|---|---|
| `hierarchical` | Splits on H1/H2 headers first, then by character limit — keeps rule sections coherent |
| `fixed_size` | Naive character-limit splits with overlap — baseline for comparison |

Each chunk is optionally enriched with a 1-2 sentence context summary generated by a small local LLM (`qwen3:1.7b`) before embedding — a technique similar to Anthropic's contextual retrieval.

**Embedding & retrieval**

Two embedding backends are supported and compared in eval:

| Model | Backend |
|---|---|
| `all-MiniLM-L6-v2` (default) | ChromaDB built-in |
| Any Ollama model | Ollama embeddings API |

Chunks are stored in ChromaDB (one collection per game). At query time, the top-k chunks are retrieved by cosine similarity, formatted with page numbers, and injected into the generation prompt.

**Generation**

A multi-turn chat chain built with LangChain. The LLM (`gemma4:e4b`) is instructed to answer solely from the retrieved excerpts and cite page numbers, or explicitly say the answer isn't in the rulebook.

**Evaluation**

A CLI eval pipeline (`python -m rag_pipeline.eval`) scores each chunking × embedding configuration using [DeepEval](https://github.com/confident-ai/deepeval) metrics — contextual precision, faithfulness, and answer relevancy — against a hand-authored question/answer dataset. Results are written to an HTML report.

## Stack

- **[LangChain](https://python.langchain.com/)** — chain orchestration
- **[Ollama](https://ollama.com/)** — local LLM inference (generation + embeddings + eval judge)
- **[ChromaDB](https://www.trychroma.com/)** — vector store
- **[LlamaCloud](https://cloud.llamaindex.ai/)** — PDF parsing
- **[DeepEval](https://github.com/confident-ai/deepeval)** — RAGAS evaluation metrics
- **[Chainlit](https://chainlit.io/)** — chat UI

## Getting started

Requires [uv](https://docs.astral.sh/uv/) and [Ollama](https://ollama.com/) running locally.

```bash
uv sync
uv run dev        # starts the Chainlit dev server
```

### Ollama models

The pipeline uses two local Ollama models. Pull them before running:

| Role | Default model | Pull command |
|---|---|---|
| Chat / generation | `gemma4:e4b` | `ollama pull gemma4:e4b` |
| Chunk context summarisation | `qwen3:1.7b` | `ollama pull qwen3:1.7b` |

## Quality checks

[Ruff](https://docs.astral.sh/ruff/) (lint + format) and [basedpyright](https://docs.basedpyright.com/) (type checking) run automatically on every commit via [pre-commit](https://pre-commit.com/).

Install the hooks after cloning:

```bash
uv sync
uv run pre-commit install
```

Run the checks manually at any time:

```bash
uv run ruff check .       # lint
uv run ruff format .      # format
uv run basedpyright       # type check
```
